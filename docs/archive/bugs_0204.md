# Bug Fixes — 02/04/2026

## Bugs Criticos

### 1. Race condition en incremento de promo codes
**Problema:** `complete-registration/route.ts` usaba patrón read-then-write (SELECT current_uses → UPDATE con valor +1). Dos registros simultáneos podían leer el mismo valor y perder un incremento.

**Fix:**
- Creada migración `040_atomic_promo_increment.sql` con función PostgreSQL `increment_promo_code_uses(p_promo_code_id UUID)` que hace `UPDATE ... SET current_uses = current_uses + 1` atómicamente.
- Actualizada `complete-registration/route.ts` para usar `supabase.rpc("increment_promo_code_uses", ...)` en lugar del patrón read-then-write.
- Tests actualizados para verificar la llamada RPC.

**Archivos modificados:**
- `supabase/migrations/040_atomic_promo_increment.sql` (nuevo)
- `apps/web/app/api/complete-registration/route.ts`
- `apps/web/app/api/complete-registration/route.test.ts`

### 2. URLs hardcodeadas e inconsistentes
**Problema:** 3 archivos tenían URLs de Vercel hardcodeadas con nombres inconsistentes (`fit-os-web` vs `fitos-web`).

**Fix:** Todas las instancias ahora usan `process.env.NEXT_PUBLIC_BASE_URL` con fallback al valor original.

**Archivos modificados:**
- `apps/web/app/layout.tsx` (línea 32)
- `apps/web/app/page.tsx` (línea 17)
- `apps/web/app/(auth)/forgot-password/page.tsx` (línea 25) — también corregida inconsistencia `fitos-web` → `fit-os-web`

### 3. Error handling en complete-registration
**Estado:** Revisado. El promo code increment ya era non-blocking (retorna 200 aunque falle). Esto es correcto por diseño: el registro del cliente es lo importante, el incremento del contador de promo es secundario. Con el nuevo RPC atómico, el error handling se simplificó (una sola llamada en vez de dos).

### 4. Sentry con sendDefaultPii: true
**Problema:** `sendDefaultPii: true` en `apps/mobile/App.tsx` enviaba datos personales (IPs, emails, cookies) a Sentry sin consentimiento explícito del usuario, violando GDPR.

**Fix:** Cambiado a `sendDefaultPii: false` con comentario explicativo.

**Archivos modificados:**
- `apps/mobile/App.tsx` (línea 40)

### 5. Error handling faltante en pantallas mobile
**Problema:** Varias queries en mobile no destructuraban `error` ni notificaban al usuario.

**Fixes:**
- `CaloriesScreen.tsx`: `loadTodayLogs()` ahora destructura error y muestra Alert.
- `AppointmentsScreen.tsx`: queries de `trainer_clients` y `profiles` ahora logean errores (no bloqueante). `handleCancel` ahora verifica error y muestra Alert si falla.
- `ProgressScreen.tsx`: ya tenía error handling correcto (verificado, no requirió cambios).

**Archivos modificados:**
- `apps/mobile/src/screens/CaloriesScreen.tsx`
- `apps/mobile/src/screens/AppointmentsScreen.tsx`

## Bugs Medios

**Problema:** `getZoneStroke()` retornaba `rgba(0,229,255,0.15)` con comentario "DEBUG: visible outline for all zones" para zonas sin molestias, mostrando un contorno cyan visible.

**Fix:** Cambiado a `"transparent"` (consistente con mobile).

**Archivos modificados:**
- `apps/web/components/health/AnatomyMap.tsx` (línea 73)

### 7. Reducción de any types en mobile

**Fix:** Reducidas a ~5 (las restantes son casts de estilo RN que requieren `as any`). Cambios:

- `useRoutineScreen.ts`: Reemplazados `as any` por tipos correctos (`SetEntry["type"]`, `{ day_label: string; week_number: number }`, `PreviousSet & {...}`).
- `ActiveTraining.tsx`: Eliminados 3 `any` en `shouldShowRpe` (los tipos ya tenían `target_rpe`).
- `RegistrationMode.tsx`: Eliminados 3 `any` en `shouldShowRpe`.

**Archivos modificados:**
- `apps/mobile/src/lib/widget-data.ts`
- `apps/mobile/src/screens/routine/useRoutineScreen.ts`
- `apps/mobile/src/screens/routine/ActiveTraining.tsx`
- `apps/mobile/src/screens/routine/RegistrationMode.tsx`

### 8. .env.example creados
**Problema:** No existían archivos `.env.example` documentando las variables requeridas.

**Fix:** Creados en ambas apps con todas las variables documentadas y agrupadas por obligatoriedad.

**Archivos creados:**
